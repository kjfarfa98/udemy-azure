# Azure 基礎知識メモ

## 用語・概念

### Azure Portal（ポータル）
- AWSの「マネコン（Management Console）」に相当
- ブラウザからGUIでリソースを管理するWebインターフェース
- 日常的に「ポータルで確認して」「ポータルからデプロイして」と呼ぶ

**AWSとの違い**
- カスタマイズ可能なダッシュボードが標準で充実（ウィジェットの追加・配置変更など）
- AWSは各サービスのコンソールが独立している感が強いのに対し、Azure Portalは統一感のあるUIで全リソースを横断管理しやすい設計
- 検索バーからリソース名・サービス名で直接飛べる点は共通

**Portal の設定**
- 言語設定はヘッダの設定アイコン（歯車）から変更可能

---

## AzureとAWSの設計思想の違い

| | Azure | AWS |
|---|---|---|
| 設計思想 | まとめて管理（エンタープライズ寄り） | 独立したサービス（マイクロサービス寄り） |
| 歴史的背景 | Microsoftのオンプレ文化（1サーバーで複数機能）をクラウドに持ち込んだ | Amazonの社内インフラをサービス化。「1つのことをうまくやる」思想 |
| メリット | 認証・課金が一元管理で楽 | サービスごとに細かく権限・スケールを制御できる |
| デメリット | アカウント単位のリスクが大きい | サービスが増えると管理が煩雑 |

この設計思想の違いはストレージだけでなく、Azure全体に通じる特徴。

## 無料試用版（Free Trial）共通の注意事項

- クォータは**リージョン単位で独立して管理**されている
- クレジット（$200/30日）が残っていてもクォータ制限は別管理なので注意
- 制限がエラーコード（`NotAvailableForSubscription` など）で明示される（AWSの無料枠と違う点）
- **リージョンを変えると解決することがある**（例：Japan East → NG、Japan West → OK）
- 人気リージョンほど無料枠が絞られる傾向あり
- 根本解決は従量課金（Pay-As-You-Go）へのアップグレード（$200クレジットは引き継がれる）

## 管理・構成

### リソースグループ（Resource Group）
- AWSに直接対応するものがない概念（サブネットとは全く別物）
- VMやストレージ・ネットワークなど、あらゆるリソースを束ねる**論理的な入れ物**
- ネットワーク的な意味はまったくない
- 同じリソースグループ内に異なるリージョンのリソースを混在できる
- リソースグループを削除すると中のリソースをまとめて削除できる

**AWSで近い概念**

| 用途 | AWSで近いもの |
|---|---|
| まとめて削除 | CloudFormationスタック |
| コスト管理 | Cost Allocationタグ |
| 権限管理 | IAMポリシー + タグベースのアクセス制御 |

AWSではタグやCloudFormationスタックで代替していたものが、Azureではリソースグループという**第一級の概念**として存在している。

**AWSとAzureのアカウント概念の対応**

| AWS | Azure |
|---|---|
| AWSアカウント（root/管理） | Azureアカウント（Microsoft/Entra IDのログインID） |
| AWSアカウント（member） | サブスクリプション |
| AWS Organizations | 管理グループ（Management Group） |

- **Azureアカウント**：ログインするためのID（MicrosoftアカウントやEntra ID）
- **サブスクリプション**：課金・リソースの管理単位（AWSのアカウントに相当）
- 1つのAzureアカウントで複数のサブスクリプションを持てる

```
Azureアカウント（1つのログインID）
├── サブスクリプション: myapp-dev
├── サブスクリプション: myapp-staging
└── サブスクリプション: myapp-prod
```

**AWSとの大きな違い**
- AWSは環境ごとにアカウントを分離する（別メールアドレス・別ログイン）
- Azureは**1つのログインIDのまま、サブスクリプションを切り替えて**環境を分離できる
- → Azureのほうが運用上シンプル

**Azureの階層構造**
```
管理グループ（Management Group）　← 複数サブスクリプションをまとめる
└── サブスクリプション　　　　　　← AWSのアカウントに相当
    └── リソースグループ
        └── リソース
```

**実務での環境分離のベストプラクティス**

| 分離レベル | 手段 | 用途 |
|---|---|---|
| 本番・非本番の分離 | **サブスクリプションを分ける** | AWSのアカウント分離と同じ考え方。これが基本 |
| 同一環境内の機能分離 | リソースグループを分ける | PJ単位・ライフサイクル単位など |
| 大規模・複数チーム | 管理グループでサブスクリプションをまとめる | 複数PJや組織をまたいだ管理 |

- リソースグループだけで本番・非本番を分けるのは**アンチパターン**（学習・検証用途のみ許容）
- AWSのタグはラベルに過ぎず削除操作などには使えないが、Azureのリソースグループはより**強い管理単位**として機能する

## コンピューティング

### 仮想マシン（Virtual Machine / VM）
- AWSの「EC2」に相当
- クラウド上に仮想サーバーを立てるサービス

**AWSとの違い**
- インスタンスタイプの命名：AWSは `t3.micro`、Azureは `Standard_B1s` のような形式
- **停止時の課金に注意**：AWSはインスタンス停止で課金停止。Azureは「停止」だけでは課金が続く。**「割り当て解除（Deallocate）」** にしないと課金が止まらない
- OSディスク：VM削除時にディスクが残る場合があり、ディスク課金が続くので削除時に要確認
- パブリックIP：AWSのElastic IPに相当するのが「パブリックIPアドレス」リソース。どちらも独立したリソースとしてVMにアタッチする形

**無料試用版での注意**
- `NotAvailableForSubscription` エラーが出たらリージョンを変えてみる
- 学習用おすすめサイズ：`Standard_D2als_v6`（コストが低く汎用タイプ）
- 使い終わったら必ず**割り当て解除（Deallocate）**すること

---

### App Service
- AWSの「Elastic Beanstalk」が最も近い
- コード（またはコンテナ）をデプロイするだけでサーバー管理不要で動かせるPaaS

**AWSの各サービスとの対応**

| App Serviceの使い方 | AWSで近いもの |
|---|---|
| コードをそのままデプロイ（Web Apps） | Elastic Beanstalk |
| コンテナをデプロイ | App Runner |
| 関数単位で動かす | Lambda ※Azureでは別途 Azure Functions がある |

**FargateはApp Serviceとは別物**
- FargateはECSのコンテナ実行基盤（インフラ管理不要のコンテナランタイム）
- AzureでFargateに近いのは **Azure Container Instances（ACI）** や **Azure Kubernetes Service（AKS）**
- App ServiceはそれらよりPaaSレイヤーが高い（インフラを意識しない）

**App Serviceのポイント**
- **App Service Plan** がセットで必要 — サーバーのスペック・料金プランに相当（EC2のインスタンスタイプ選択に近い感覚）
- 複数のアプリを1つのApp Service Planで動かすことができる
- スロット機能（Blue/Greenデプロイ）が標準で使える

**無料試用版での注意**
- Free・Basic・Premiumすべてのプランがクォータ0になる場合がある（プラン変更では解決しない）
- リージョンを変えると解決することがある（例：Japan East → NG、Japan West → OK）
- 本番運用では Japan East が推奨（データセンターが新しくサービス提供が早い）
- 使い終わったらApp Serviceを削除してクレジットの消費を止める

## ストレージ

### ストレージアカウント（Storage Account）
- AWSの「S3」より広い概念。1つのアカウントに複数のストレージサービスが同居する
- 認証・課金・アクセスキーはストレージアカウント単位で一元管理
- 注意：アクセスキーが漏れると全サービスへのアクセスが危険にさらされる
- 用途やセキュリティ要件に応じてストレージアカウントを分けることも可能
- ストレージアカウント名はグローバルで一意

| ストレージアカウント内のサービス | AWSで近いもの |
|---|---|
| **Blob Storage** | S3（オブジェクトストレージ） |
| **File Storage** | EFS（マネージドファイル共有） |
| **Queue Storage** | SQS（メッセージキュー） |
| **Table Storage** | DynamoDB（NoSQLテーブル）※簡易版 |

**構造イメージ**
```
Azureサブスクリプション
└── ストレージアカウント（S3+EFS+SQS+DynamoDBをまとめた中間層。直接対応するAWSリソースなし）
    └── コンテナ（≒ S3バケット）
        └── Blob（≒ S3オブジェクト）
```

- URLの構造：`https://<storageaccount>.blob.core.windows.net/<container>/<blob>`
- **Blob Storageは独立したリソースではない** — ストレージアカウントを作った時点で使えるサービスエンドポイント
- コンテナ = S3バケット（Dockerのコンテナとは全くの別物なので注意）
- ディレクトリは存在せず、Blob名に `/` を含めた仮想パスで階層を表現（S3と同じ）

**Blobとは**
- **Binary Large OBject** の略
- バイナリ形式の大きなデータ（画像・動画・ログなど）を格納するストレージという意味
- AWSがS3（Simple Storage Service）とシンプルな名前にしたのに対し、Azureは技術用語をそのまま使っている
